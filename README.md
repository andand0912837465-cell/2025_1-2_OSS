##  Week 10
1. OSS의 정의 및 의미
OSS (Open Source Software): 누구나 소스 코드를 자유롭게 보고(공개), 사용, 수정, 배포할 수 있는 라이선스를 만족하는 소프트웨어입니다.
핵심: 단순히 '무료'를 넘어, 코드의 자유(Freedom)를 보장하는 것을 목표로 합니다.
OSI (Open Source Initiative): 공개 소스 정의(OSD)를 관리하고 인증 마크를 부여하는 비영리 조직입니다.
자유 소프트웨어 vs. 오픈소스: 자유 소프트웨어는 사용자의 자유를 강조하며 철학적 측면이 강하고, 오픈소스는 실용적이고 기술적인 측면을 강조합니다.
2. OSS 주요 라이선스 
GPL (General Public License)
Copyleft (강함)
있음 (무조건)
GPL 코드를 사용한 2차 저작물은 무조건 소스 코드 공개 및 GPL 라이선스 유지 의무.
LGPL (Lesser GPL)
Copyleft (중간)
있음 (조건부)
라이브러리로 동적 링크하여 사용 시 애플리케이션 코드는 공개하지 않아도 됨.
Apache License 2.0
Permissive (약함)
없음
특허권(Patent) 사용 권한을 부여하며, 라이선스 사본과 저작권 고지사항만 포함하면 됨.
MIT License
Permissive (가장 약함)
없음
최소한의 의무(저작권 명시)만 지키면 수정, 배포, 상업적 이용 모두 자유로움.
임시 저장 (Stash) 개요
목적: 커밋하지 않은 현재 작업 내용(Modified 파일)을 별도의 스택에 저장하여, 작업 디렉터리를 깨끗하게 비우는 기능입니다. (긴급 브랜치 전환 시 사용)
영향 영역: 작업 디렉터리(Working Directory)와 스테이징 영역(Staging Area)의 변경 내용을 저장합니다.
Untracked 파일 삭제 (git clean)
git clean -n (Dry-Run): 삭제될 파일 목록만 미리 확인합니다.
git clean -f (Force): Untracked 파일을 강제로 삭제합니다.
git clean -i (Interactive): 삭제할 파일을 대화형 모드로 선택하여 삭제합니다.
##  Week 11
다양한 브랜치 병합 (Merge)
1. Fast-forward 병합
조건: 메인 브랜치가 분기 이후 추가 커밋이 없어 이력이 일렬로 이어져 있을 때.
결과: 별도의 병합 커밋 없이, 브랜치 포인터만 최신 커밋으로 이동합니다. (빨리 감기)
2. 3-way 병합 (3방향 병합)
조건: 메인 브랜치와 서브 브랜치 모두 독립적인 커밋을 만들어 이력이 갈라진 상태일 때.
결과: 공통 조상 커밋을 기준으로 변경 사항을 통합하고, 그 결과를 반영하는 새로운 병합 커밋(Merge Commit)을 생성합니다.
옵션: it merge --no-ff [브랜치명]를 사용하면 Fast-forward가 가능한 상황에서도 강제로 3-way 병합을 수행하여 병합 이력을 남깁니다.
3. Squash 병합 (--squash)
목적: 서브 브랜치에서 만든 여러 개의 커밋을 하나의 커밋으로 압축하여 메인 브랜치에 통합합니다.
명령: git merge --squash [브랜치명]
결과: 변경 내용이 스테이징 영역에 놓이며, 사용자가 **직접 git commit**을 실행해야 최종적으로 압축된 커밋이 생성됩니다.
4.병합 충돌 발생
원인: 두 브랜치에서 같은 파일의 같은 라인을 서로 다르게 수정하여 Git이 자동 통합을 할 수 없을 때 발생합니다.
확인: git merge 명령 실행 후 CONFLICT (content) 메시지가 출력됩니다.
5. 충돌 해결 과정
식별: 충돌 파일 내부에 <<<<<<< HEAD, =======, >>>>>>> [브랜치 이름] 마커가 표시됩니다.
수정: 개발자가 직접 코드를 선택/조합하고, 모든 마커를 제거합니다.
해결 표시: 수정된 파일을 git add [파일 이름] 명령으로 스테이징 영역에 올려 해결되었음을 알립니다.
완료: **git commit**을 실행하여 병합 커밋을 생성하고 병합을 마무리합니다.
충돌 포기: 병합 시도를 취소하고 싶을 때는 git merge --abort 명령을 사용합니다.
##  Week 12
브랜치 리베이스
1. Rebase 개요
목적: 브랜치의 기준점(Base)을 다른 커밋으로 옮겨 심어, 히스토리를 병합 커밋이 없는 일직선(Linear) 형태로 만듭니다.
명령: git switch [서브 브랜치] 이동 후 git rebase [메인 브랜치]
작동 방식: 서브 브랜치 커밋들을 복사하여 메인 브랜치의 최신 커밋 위에 순서대로 붙여 넣고, 커밋 ID는 새로 생성됩니다.
2. Rebase 주의사항
공유된 커밋 금지: 이미 원격 저장소에 Push 하여 공유된 커밋에 대해 Rebase를 하면 다른 개발자와 이력 불일치로 심각한 문제가 발생할 수 있습니다. 개인 작업 브랜치에서만 사용해야 합니다.
3. 최신 커밋 수정 (--amend)
목적: 가장 최근 커밋의 메시지를 수정하거나, 빠뜨린 내용을 추가할 때 사용합니다. (새 커밋 ID 생성)
사용법 (내용 수정): 파일 수정 -> git add [파일] -> git commit --amend
4. 과거 여러 커밋 수정 (rebase --interactive, -i)
명령: git rebase -i HEAD~[숫자]
목적: 지정된 숫자만큼 과거 커밋들의 순서, 내용, 메시지 등을 조작할 수 있는 대화형 모드입니다.
주요 대화형 명령:
pick (p): 커밋을 그대로 사용 (기본).
reword (r): 커밋 내용은 유지하고 메시지만 수정합니다.
squash (s): 현재 커밋을 바로 이전 커밋과 합쳐서 하나의 커밋으로 통합합니다.
drop (d): 해당 커밋을 삭제합니다.
